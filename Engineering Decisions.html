JackpotBet Admin Statistics API

Engineering Decisions

A record of the architectural choices, library selections, and design rationale behind this Go REST API — covering what was built, why each decision was made, and what trade-offs were accepted.

Language Go 1.23
Database MongoDB + Redis
Pattern Handler → Service → Repository
Scope Read-heavy internal admin API
A note on authorship

Every architectural decision, design pattern, and engineering trade-off in this codebase was conceived and reasoned through by me — the three-layer structure, the cursor pagination approach, the $setWindowFields + $facet rank pipeline, the materialized-view analysis, the UTC enforcement points, all of it. Where I knew what I wanted, I designed it first and directed the implementation.

That said, some parts of the code were written or polished with AI assistance: inline code comments, boilerplate sections of repetitive handler code that follow an already-established pattern, and the prose in this document. AI in those cases was a faster keyboard, not an architect — it built within the boundaries I had already drawn, and did not decide what those boundaries should be.

I want to be transparent about this rather than claim every line was typed by hand. The approaches are mine; the execution, in places, had help.

01 · Architecture

Three-Layer Design

The project uses a strict Handler → Service → Repository layering with no domain events, adapters, or CQRS. This was a deliberate choice calibrated to the problem's actual complexity.

Handler (api/)
→
Service (service/)
→
Repository (repository/)
HTTP parsing · routing · validation · JSON encoding
Business logic · grouping · arithmetic · caching
MongoDB queries · aggregation pipelines · indexes
Decision
Why not a simpler single-layer CRUD structure?

The stat endpoints require MongoDB aggregation pipelines — $group, $setWindowFields, $facet. Those pipelines need a dedicated home that isn't a handler. The repository layer provides that. A CRUD-only project could skip it, but this one couldn't.

Kept
Service layer for grouping logic (groupDailyWagersByDate), decimal arithmetic, and cache coordination — things that have no place in a handler or a raw DB call.
Decision
Why not add domain/event layers?

Domain events make sense when actions trigger side-effects across bounded contexts. This API has one context (transaction statistics) and all callers are internal. Adding an event bus or domain layer would be indirection without payoff — more surface area for bugs with no architectural benefit at this scale.

Trade-off
Introducing real-time notifications or cross-service communication later would require adding an event layer. The current design would need refactoring at that point — that cost was accepted intentionally.
02 · Library Choices

Frameworks and Dependencies

HTTP Router
gorilla/mux over Gin, Echo, or Fiber

Gin and Echo are full frameworks that wrap net/http in their own context types. Any middleware or handler written for them is not portable to standard net/http. Since validation is handled by go-playground/validator directly — not by the framework — Gin provides no validation benefit and locks the codebase into its context.

gorilla/mux is a router only. Handlers receive standard http.ResponseWriter and *http.Request. Path variables come from mux.Vars(r), subrouters handle auth middleware grouping cleanly, and nothing about the code is framework-specific.

Trade-off
No built-in rate limiting, no automatic swagger generation. Those were out of scope; they could be added as middleware or a separate tool without changing any handler code.
Config
viper over burntsushi/toml

Viper has native multi-path search: it finds conf/default.toml whether the binary is run from the repo root or from cmd/seed/, without any manual path construction. It also supports naming different environments by file name (test.toml, production.toml) with a single LoadFrom("name") call.

Trade-off
Viper silently ignores unknown keys; burntsushi/toml errors on them. Unknown keys in config are real mistakes and stricter validation would catch them earlier. This was accepted because the config structure is small and explicit.
Decimal Arithmetic
shopspring/decimal throughout the service layer

Monetary amounts and crypto values cannot use float64 — floating point rounding errors compound across summation. bson.Decimal128 stores values precisely in MongoDB but has no arithmetic operations. The boundary is explicit: Decimal128 is parsed to a decimal string at the repository edge, the service uses shopspring/decimal for all arithmetic, and results are formatted with .StringFixed(n) before returning.

Logging
zerolog behind a Logger interface

All service and handler code receives a logger.Logger interface, not a concrete zerolog logger. This means the logger can be swapped for testing, replaced with a different backend, or silenced entirely — without changing any call site. zerolog is fast and structured by default, which is what a JSON-log-parsing infrastructure (ECS, Datadog, etc.) needs.

03 · HTTP Layer

Handler Design and Error Handling

Every handler follows the same sequence. Deviating from it would mean inconsistent validation or error shape across endpoints.

  • Decode input — DecodeQuery for GET (via qs tags), DecodeJSONBody for POST
  • Apply defaults — e.g. if query.Limit == 0 { query.Limit = 20 }
  • Validate with go-playground/validator — wraps the error string into an AppError
  • Call query.DateRangeFilter.Validate() — normalises From/To to UTC, checks From < To
  • Convert raw values — e.g. hex cursor string to bson.ObjectID
  • Call the service, respond with a.respond or a.respondError

Structured errors

All error responses share one shape: { "code": "VALIDATION_ERROR", "message": "..." }. The AppError type carries the HTTP status, the code, and the message. Service-layer errors that are not AppError become 500 INTERNAL_ERROR automatically — handlers never manually produce 500s.

Auth middleware

A static token check on the /internal subrouter. The middleware reads Authorization: <token>, compares with the configured value, and rejects with 401 before the handler runs. The middleware is isolated — replacing it with JWT or mTLS later means changing one file.

04 · Startup Safety

Nil-Field Validation on Boot

A common mistake in dependency injection: add a new field to Repos or Services, forget to wire it in init.go, and ship a binary that panics at runtime when that field is first accessed — possibly in production, under load.

Mechanism
Reflect-based nil check on every Repos and Services field at startup

server.validate() walks every field of Repos and Services using reflection and fatals immediately if any pointer or interface field is nil. It also checks for typed nils — the case where an interface field is non-nil but wraps a nil concrete pointer, which would also panic at the first method call.

The result is that a missing wire in init.go fails fast and loudly at startup rather than silently at runtime.

The compile-time complement is the interface satisfaction check at the bottom of each mongo implementation file:

// Fails at compile time if TransactionRepository is missing any interface method.
var _ repository.TransactionRepository = (*TransactionRepository)(nil)
05 · Data Seeder

Three-Stage Concurrent Pipeline

The seeder generates 4 million transaction documents (2 million rounds × 2 transactions each). A naive sequential loop would be far too slow. The seeder is structured as a three-stage pipeline where stages run concurrently and communicate through channels.

Scheduler
Stage 1 · 1 goroutine
→
Builder
Stage 2 · N goroutines
→
Writer
Stage 3 · N goroutines
  • Scheduler — slices 2M rounds into 500-round batches and sends job descriptors over a channel. No data is generated here.
  • Builder — CPU-heavy. Multiple workers run in parallel, each with its own rand.Rand (the global rand is not goroutine-safe without a lock). Generates random amounts, currencies, and timestamps for each batch, producing 1000 documents ready to insert.
  • Writer — I/O-heavy. Multiple workers fire concurrent InsertMany calls to MongoDB. Workers are capped at min(runtime.NumCPU(), 20) — beyond 20, connection overhead and channel contention outweigh concurrency gains.

All timestamps are generated as UTC (time.Now().UTC()) spread randomly over a one-year window, so the seeded data has realistic temporal distribution for testing the date-range stat endpoints.

06 · API Design

Endpoints and Key Decisions

MethodPathDescription
GET /internal/transactions Paginated list with optional date filter
POST /internal/transactions Create a single Wager or Payout transaction
GET /internal/gross_gaming_rev GGR per currency + USD over a date range
GET /internal/daily_wager_volume Daily wager volume grouped by date with per-currency breakdown
GET /internal/user/{userId}/wager_percentile User's wager rank and top-N% position in the period

Cursor-based pagination for GET /transactions

Decision
ObjectID cursor pagination over offset/limit

Offset pagination (SKIP n) requires MongoDB to scan and discard n documents on every page request. At 4 million documents this is expensive and gets slower as the offset increases. Cursor pagination passes the last seen ObjectID and uses { _id: { $gt: cursor } } — the index seek is O(log n) regardless of how deep into the result set the client is.

Trade-off
Cannot jump to an arbitrary page number. For an admin API doing sequential exports this is not a limitation — pages are consumed in order.

Grouped response for GET /daily_wager_volume

Decision
Map-keyed currencies, not a fixed struct with named fields

An initial design used a struct with named currency fields: { ETH: ..., BTC: ..., USDT: ... }. This was rejected because it hardcodes the currency set into both the API response and the frontend parsing code. New currencies would require a code change on both sides. The response uses map[string]CurrencyVolume instead — adding a new currency requires only a new entry in the rate table.

Grouping (flat (date, currency) rows from MongoDB → one entry per date) happens in a pure helper function groupDailyWagersByDate using a []string slice to preserve date order alongside a map for O(1) lookup.

07 · Wager Percentile

Rank Computation Without Fetching All Users

The naive approach to GET /user/{userId}/wager_percentile is to fetch all users' wager totals, sort them in application code, and find the target user's position. This transfers potentially thousands of documents over the wire for every request.

Decision
MongoDB computes the rank; only one document returns to the application

The aggregation uses $setWindowFields to assign a rank to each user by total USD wagered (descending), then a $facet to run two sub-pipelines in parallel on the grouped result:

$match (type=Wager + date range)
$group (by userId, sum usdAmount → one doc per user)
$facet {
  "userRank": [
    $setWindowFields { sortBy: totalUSD desc, output: { rank: $rank } }
    $match { _id: targetUserId } → 0 or 1 document
  ],
  "total": [ $count "n" ] → { n: 700 }
}

$facet always returns exactly one document. The userRank array is empty when the user had no wagers in the period — the service places them last: rank = totalUsers + 1, percentile = 100%.

The percentile formula matches the assignment example: if a user is 10th out of 500, topPercentile = 10/500 × 100 = 2.00 — meaning they are in the top 2%.

Index supporting the aggregation

A compound index { type: 1, createdAt: 1 } covers the $match stage filter on type = "Wager" and the date range, so MongoDB can satisfy the match from the index without a collection scan.

08 · Caching

Redis TTL Caching and Why Not a Materialized View

The GGR, daily wager volume, and wager percentile endpoints all aggregate over millions of transactions. Running the full pipeline on every request is expensive at any meaningful scale. A caching layer was added.

Decision
Simple Redis TTL caching, not a materialized view

A materialized view would pre-compute daily totals into a daily_stats MongoDB collection via a nightly cron, making GGR and daily wager volume O(days) reads instead of O(millions). This is the right long-term architecture.

It was intentionally not built here because it introduces coordinated complexity: the cron must handle backdated transactions (manually inserted corrections), cache invalidation must be coordinated between the nightly recompute and any manual correction endpoint, and the recompute window (how many past days to redo) requires operational tuning. That complexity exceeds the scope of this assignment.

Instead
TTL-based Redis caching. Each unique (from, to) date range maps to a JSON blob with a configurable TTL (default 24h). Stale data is acceptable within the TTL window for an admin reporting API.

Cache key design

Dates are normalised to UTC day strings so queries spanning the same calendar days share one entry regardless of the time-of-day component:

Redis memory management

On connect, the application sets maxmemory-policy allkeys-lru so Redis evicts the least-recently-used cache entry when memory is full — rather than erroring on new writes. An optional max_memory config value (e.g. "256mb") caps total Redis memory; leaving it empty delegates memory management to the operator's Redis configuration.

09 · Time Handling

UTC Throughout

All timestamps are stored and returned in UTC. There are three enforcement points:

  • Query params — DateRangeFilter.Validate() has a pointer receiver and normalises From and To to UTC in-place before any handler calls the service. Clients can send any timezone offset; it is converted at the boundary.
  • Persistence — time.Now().UTC() is the only form used when generating a createdAt timestamp. Caller-supplied times (input.CreatedAt) are normalised with .UTC() before storage.
  • Seeder — the historical time window is computed from time.Now().UTC() so all 4 million seeded timestamps are UTC.

Elapsed-time measurements used only for logging duration (time.Now() / time.Since() in middleware) are exempt — they produce durations, not stored timestamps.

10 · Interface Design

Repository Interfaces and Swappability

The repository/ package contains only interfaces and plain-Go filter/result types — no MongoDB imports. The concrete implementations live in repository/mongo/. This means:

  • Tests can inject a mock that satisfies the interface without starting MongoDB.
  • Switching to a different database is a new implementation file, not a rewrite of service logic.
  • The service layer never sees a bson.D or a mongo.Cursor — all amounts cross the boundary as decimal strings, all IDs as bson.ObjectID (the one bson type allowed at the interface boundary as a cursor identifier).

The same isolation applies to the logger: all packages receive a logger.Logger interface. The zerolog implementation is wired in server/init.go and is invisible to anything that uses it.
